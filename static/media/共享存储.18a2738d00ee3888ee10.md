共享存储的核心不是“容器把文件同步到云盘”，而是：**容器里的 `/root/.codex` 被挂载成了一个外部文件系统入口**。服务写这个目录时，实际写到的是挂载在宿主机或 Kubernetes 节点上的 FileNAS。

以火山引擎 **弹性文件存储 FileNAS** 为例，可以这样理解。火山引擎文档中心把“弹性文件存储”列在存储产品下，旁边还有对象存储、弹性块存储、文件存储 vePFS 等产品分类。FileNAS 这类 NAS 适合提供多客户端共享的文件语义；Docker 官方也说明，容器持久化通常用 volume 或 bind mount，volume 可以被多个容器共享，并且一些 volume driver 支持 NFS 这类外部共享存储。Docker 文档还特别提到，写入容器自身可写层会走 storage driver，而写入 volume 更接近直接写宿主文件系统。参考：[Docker Volumes](https://docs.docker.com/engine/storage/volumes/) 、[Docker Bind Mounts](https://docs.docker.com/engine/storage/bind-mounts/) 、[火山引擎文档中心存储分类](https://www.volcengine.com/docs?lang=zh)。

典型链路是这样的：

```text
容器进程
  写 /root/.codex/session.json
        |
        v
容器 mount namespace 发现 /root/.codex 是挂载点
        |
        v
Docker/K8s volume 映射到宿主机路径
例如 /mnt/filenas/codex
        |
        v
宿主机上的 /mnt/filenas 是 FileNAS 的 NFS 挂载
        |
        v
Linux NFS 客户端通过 VPC 把 write/create/rename 等请求发到 FileNAS 挂载点
        |
        v
FileNAS 后端保存文件数据和元数据
```

所以，如果容器里服务写：

```bash
echo hello > /root/.codex/a.txt
```

它看起来是在写容器内部目录，但如果启动容器时做了挂载：

```bash
docker run \
  --mount type=bind,src=/mnt/filenas/codex,dst=/root/.codex \
  your-image
```

那么实际落盘位置就是宿主机的：

```bash
/mnt/filenas/codex/a.txt
```

而这个宿主机目录本身又是 FileNAS 的 NFS 挂载目录，因此数据最终由 FileNAS 保存。

多个容器共享时有两种常见形态：

```text
同一台宿主机：
container A /root/.codex -> /mnt/filenas/codex
container B /root/.codex -> /mnt/filenas/codex
```

或者多台宿主机：

```text
ECS-1: /mnt/filenas -> FileNAS
ECS-2: /mnt/filenas -> FileNAS
ECS-3: /mnt/filenas -> FileNAS

每台机器上的容器都把 /mnt/filenas/codex 挂到 /root/.codex
```

在 Kubernetes 里就是 PV/PVC 模式。Kubernetes 官方把 PersistentVolume 定义成集群中的一块存储，可以由 NFS、iSCSI 或云厂商存储实现；Pod 通过 PVC 使用它。也就是说 VKE 里通常会用 FileNAS CSI / NFS PV 把 FileNAS 挂到 Pod 内的 `/root/.codex`。

一个简化版 Kubernetes 形态：

```yaml
volumeMounts:
  - name: codex-data
    mountPath: /root/.codex

volumes:
  - name: codex-data
    persistentVolumeClaim:
      claimName: filenas-codex-pvc
```

要点是：**容器不知道 FileNAS 的存在**。它只看到一个普通 Linux 目录。真正负责把文件操作转成网络存储请求的是宿主机内核里的文件系统/NFS 客户端。

需要注意几个工程细节：

1. `/root/.codex` 一旦被挂载，镜像里原本这个目录下的内容会被挂载内容遮住。
2. 多容器同时写同一个文件，要自己处理并发，比如文件锁、原子 rename、数据库事务等。
3. NFS/共享文件系统适合共享文件、缓存、工作目录，不适合所有高并发小文件写入场景。
4. 权限看 UID/GID，不是看“容器名字”。容器里的 root 写入，映射到 NFS 上通常也是某个 Linux 用户身份。
5. 数据持久性不取决于容器生命周期。容器删了，只要 FileNAS 文件系统和挂载目录数据还在，文件还在。

一句话总结：**FileNAS 是共享文件系统，Docker/K8s 只是把它挂进容器路径；服务写 `/root/.codex` 时，写操作被内核路由到 FileNAS，而不是 Docker 事后同步。**
