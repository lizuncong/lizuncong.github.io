# Codex Workflow Authentication

## 正文

本文说明 `aigameforge-workflow` 中 Codex agent 主链路为什么以 Codex 登录态为主，服务器上如何准备登录态，以及 `auth.json` 轮换脚本在运行时承担什么职责。

## 一句话结论

`aigameforge-workflow` 的 Codex 主链路运行的是 `@openai/codex-sdk` 拉起的本地 Codex agent。它需要像一个已登录的 Codex 用户或 workspace 身份一样运行，因此主认证方式是通过 Codex 登录缓存，也就是 `CODEX_HOME/auth.json` 或等价 credential store。API key 不是完全不能用，但更适合普通 OpenAI API 调用或单次 `codex exec` 自动化，不适合作为当前 server workflow 的主身份模型。

## 当前实现事实

`server/codex-agent/runner/index.ts` 中，主链路通过 `new Codex(...)` 创建 Codex SDK client，并传入本地 Codex CLI runtime、运行环境、developer instructions、sandbox、MCP servers 和 workspace 配置。

代码仍会读取 `CODEX_API_KEY || OPENAI_API_KEY`，如果存在就透传给 `new Codex(...)`。这表示 API key 路径保留了兼容能力，但不是当前推荐的主身份方案。

每次 Codex run 前，runner 会执行 `syncCodexAuthBeforeRun(...)`，该函数触发 auth rotation hook。hook 默认调用 `scripts/rotate-auth.mjs`，用于把当前可用的 Codex 登录缓存同步到运行目标。

`scripts/rotate-auth.mjs` 的默认目标是：

```text
/root/.codex/auth.json
```

这符合容器或服务器中 root 用户运行 Codex runtime 的常见路径。实际部署也可以通过 `CONFIG_ROTATION_AUTH_TARGET` 覆盖目标路径。

## 官方认证模型

OpenAI Codex 官方文档把 OpenAI 模型下的 Codex 登录分成两类：

- Sign in with ChatGPT：使用 ChatGPT/Codex 用户或 workspace 身份。
- Sign in with an API key：使用 OpenAI Platform API key。

两者不是简单的“同一把钥匙不同写法”，而是不同身份域：

| 方式 | 身份归属 | 适合场景 |
| --- | --- | --- |
| ChatGPT 登录、`auth.json`、Codex access token | ChatGPT 用户或 Business/Enterprise workspace | Codex SDK、本地 Codex agent、需要 workspace 权限、Codex entitlement、审计和企业管控的 server workflow |
| `CODEX_API_KEY` / `OPENAI_API_KEY` | OpenAI Platform API organization | 普通 API 调用、隔离的一次性 `codex exec` 自动化、image fallback 等 API 型工具 |

官方文档还说明：

- Codex CLI / IDE / app 登录后会缓存登录信息，文件存储时位于 `CODEX_HOME/auth.json`，默认 `CODEX_HOME` 是 `~/.codex`。
- ChatGPT 登录态在使用中会自动刷新 token。
- Codex access token 面向 trusted scripts、schedulers 和 private CI runners，适合需要 ChatGPT workspace access、ChatGPT-managed Codex entitlements 或 enterprise workspace controls 的自动化。
- `CODEX_API_KEY` 在公开稳定环境变量中被定位为 `codex exec` 的单次非交互运行凭据，并建议 inline 使用，避免作为 job-level 环境变量暴露给会运行仓库代码的 workflow。

## 为什么不用 API key 做主认证

通俗讲，API key 更像 OpenAI Platform 的“付款卡”，而 Codex SDK 主链路需要的是“已登录的 Codex 用户身份”。

具体原因有 5 个。

1. 身份和权限不同。

   `auth.json` 或 Codex access token 代表 ChatGPT/Codex workspace 身份，可以继承 workspace 权限、Codex Local 权限、RBAC、企业数据保留和地域策略。API key 代表 Platform organization，不等价于 ChatGPT workspace member。

2. 额度和 entitlement 不同。

   Codex 的 workspace entitlement、ChatGPT plan credits、Codex usage limit 和企业管控通常挂在 Codex/ChatGPT 身份上。API key 走 Platform API 计费和 API organization 策略，不能自然表达“这个 server workflow 以哪个 ChatGPT workspace 用户运行”。

3. SDK 运行模型不同。

   当前 workflow 不是直接用 OpenAI SDK 调 `responses.create(...)`，而是用 `@openai/codex-sdk` 控制本地 Codex runtime，让 Codex 执行工程任务、运行命令、调用 MCP 和维护 session。这个模型更接近“服务器上有一个登录过的 Codex agent”，而不是“服务端拿 API key 发一个 HTTP 请求”。

4. 安全暴露面不同。

   Codex agent 会运行仓库中的命令、测试和工具。如果把 `OPENAI_API_KEY` 或 `CODEX_API_KEY` 长期放在整个服务进程或 job 环境里，仓库脚本、测试、依赖生命周期 hook 或被攻击的工具都有机会读取它。官方对自动化也特别提醒，不要把这些 key 作为 job-level env 暴露给会 checkout 或运行 repository-controlled code 的 workflow。

5. 运维能力不同。

   `auth.json` 可以通过受控目录、状态文件、启动同步和失败轮换管理。当前仓库已经有 `rotate-auth.mjs` 和 `auth-rotation-hook.ts`，能在 startup/sync 时同步当前登录缓存，在 auth/quota 失败后标记当前文件并切到下一个可用文件。API key 虽然也能轮换，但不能解决 workspace 身份、Codex entitlement 和 ChatGPT 管控归属问题。

## 服务器登录方式

推荐的服务器登录方式是：在服务启动前或启动时准备 Codex 登录态，业务请求进来时只复用登录态，不在请求路径里做人机登录。

### 方式 A：Codex access token 登录

适用条件：

- ChatGPT Business 或 Enterprise workspace。
- workspace owner/admin 已允许创建 access token。
- 如果 token 要用于 Codex Local，还需要允许成员使用 Codex Local。

创建 token 的入口是 ChatGPT 管理后台：

```text
https://chatgpt.com/admin/access-tokens
```

服务器启动时执行：

```bash
export CODEX_HOME=/root/.codex
mkdir -p "$CODEX_HOME"
printf '%s' "$CODEX_ACCESS_TOKEN" | codex login --with-access-token
```

执行后，Codex 会把 agent identity credential 写入 auth storage。文件模式下就是：

```text
/root/.codex/auth.json
```

之后 Node 服务中的 Codex SDK 正常创建：

```ts
const codex = new Codex();
const thread = codex.startThread(options);
```

业务代码不需要自己解析 token，也不应该把 token 写入日志或返回给前端。

### 方式 B：预先生成 auth.json 后注入服务器

适用场景：

- 服务器没有交互浏览器。
- 还没有 access token 权限，但能在可信机器完成 `codex login`。

流程：

1. 在可信机器上运行 `codex login` 或 `codex login --device-auth`。
2. 确认生成 `CODEX_HOME/auth.json`。
3. 通过 secret manager、Kubernetes secret、镜像启动脚本或其他受控方式注入服务器。
4. 确保服务器运行 Codex 时的 `CODEX_HOME` 指向包含该 `auth.json` 的目录。

`auth.json` 等同密码，不能提交到 Git、不能写进镜像公开层、不能贴到工单或聊天，也不能出现在日志里。

### 方式 C：API key 仅作有限 fallback

如果只是运行单次 `codex exec`，或者某个 MCP/tool 要调用普通 OpenAI API，可以使用 API key。

例如当前 image asset fallback 工具会读取 `OPENAI_API_KEY` 或 `CODEX_API_KEY` 调 OpenAI image API。这类用途是普通 API 调用，不等同于 Codex SDK 主身份。

API key 自动化应遵循两个原则：

- 尽量只在单次命令调用范围内设置，不放成全局长期环境变量。
- 不和不可信仓库代码、依赖 hook、测试脚本共享同一个进程环境。

## 当前 auth rotation 机制

当前仓库的 auth rotation 由两部分组成。

`server/codex-agent/config-rotation/auth-rotation-hook.ts` 负责判断是否启用轮换，以及在 startup、sync、auth failure、quota failure 等场景执行脚本。

只要配置了：

```text
CONFIG_ROTATION_AUTH_TSOURCE_DIR
```

hook 就会默认用当前 Node 进程执行：

```text
scripts/rotate-auth.mjs
```

也可以通过下面的变量覆盖 hook 命令和参数：

```text
CODEX_AUTH_FAILURE_HOOK_COMMAND
CODEX_AUTH_FAILURE_HOOK_ARGS
CODEX_AUTH_FAILURE_HOOK_TIMEOUT_MS
```

`scripts/rotate-auth.mjs` 负责实际同步或轮换。关键参数如下：

| 参数或环境变量 | 含义 |
| --- | --- |
| `--source-dir` / `CONFIG_ROTATION_AUTH_TSOURCE_DIR` | 存放候选 auth 文件和 `state.json` 的源目录 |
| `--target` / `CONFIG_ROTATION_AUTH_TARGET` | 要写入的目标 auth 文件，默认 `/root/.codex/auth.json` |
| `--state` / `CONFIG_ROTATION_STATE` | 状态文件路径，默认是 source dir 下的 `state.json` |
| `--current` / `CONFIG_ROTATION_CURRENT` | 初始化或修复时指定当前文件 |
| `CODEX_AUTH_FAILURE_REASON` | 轮换原因，例如 `startup`、`sync`、`auth`、`quota`、`manual` |

脚本行为可以概括为：

- `startup` / `sync`：把 `state.json` 中当前 active 文件同步到目标 `auth.json`，不主动切换到下一个候选。
- `auth` / `quota`：把当前文件标记为 failed，选择下一个 available 文件，原子替换目标 `auth.json`。
- 没有替代文件时：保留或修复当前 active 状态，并报错提示没有可用替换文件。
- 写目标文件时：使用临时文件、备份和原子替换，避免半写入 auth 文件。

源目录中的 `state.json` 用来记录哪个 auth 文件是当前 active，哪些文件 available，最近一次 replacement 是什么，以及是否有 pending replacement。

## 推荐部署形态

推荐把认证分成三个层次管理：

1. Secret manager 层。

   保存 `CODEX_ACCESS_TOKEN`，或保存一组已生成的 auth cache 文件。不要把真实 token 或 auth 文件放入仓库。

2. 启动同步层。

   容器启动时运行 `codex login --with-access-token`，或使用 `rotate-auth.mjs` 把源目录中的 active auth 文件同步到目标 `CODEX_HOME/auth.json`。

3. 运行时复用层。

   `aigameforge-workflow` 业务请求只调用 Codex SDK。SDK 通过本地 Codex runtime 读取 `CODEX_HOME/auth.json` 或 credential store，不在每个请求里重新登录。

最小启动示例：

```bash
export CODEX_HOME=/root/.codex
mkdir -p "$CODEX_HOME"
printf '%s' "$CODEX_ACCESS_TOKEN" | codex login --with-access-token
pnpm start
```

带现有轮换脚本的示例：

```bash
export CODEX_HOME=/root/.codex
export CONFIG_ROTATION_AUTH_TSOURCE_DIR=/run/secrets/codex-auth
export CONFIG_ROTATION_AUTH_TARGET=/root/.codex/auth.json
node scripts/rotate-auth.mjs --source-dir "$CONFIG_ROTATION_AUTH_TSOURCE_DIR" --target "$CONFIG_ROTATION_AUTH_TARGET" --reason startup
pnpm start
```

上面命令只是说明形态。实际生产环境应由部署系统提供 secret、目录权限、文件 owner、只读挂载和日志脱敏策略。

## 故障判断

| 现象 | 常见原因 | 处理方向 |
| --- | --- | --- |
| 访问 `https://chatgpt.com/admin/access-tokens` 回到首页 | 不是 Business/Enterprise workspace、没切到正确 workspace、管理员未开启 access token 权限或 Codex Local 权限 | 找 workspace owner/admin 开权限，确认当前浏览器账号和 workspace |
| Codex SDK 报 authentication / unauthorized | 目标 `auth.json` 缺失、过期、被写错路径或权限不可读 | 检查 `CODEX_HOME`、`CONFIG_ROTATION_AUTH_TARGET`、文件权限和 rotation 日志 |
| Codex SDK 报 usage limit / quota | 当前 ChatGPT/Codex 身份额度不足 | 触发 quota rotation，或更换有额度的 Codex workspace 身份 |
| image fallback 报 `OPENAI_API_KEY or CODEX_API_KEY is not configured` | 这是普通 API 型工具缺少 API key，不是 Codex SDK 主身份缺失 | 为 image fallback 单独配置 API key，或改回 builtin image path |
| API key 配了但 Codex workspace 权限不生效 | API key 属于 Platform API organization，不是 ChatGPT workspace 身份 | 使用 Codex access token 或 `auth.json` |

## 安全边界

必须遵守以下边界：

- 不提交 `auth.json`、access token、API key、`.env` 或 secret 文件。
- 不在日志里打印 `CODEX_ACCESS_TOKEN`、`OPENAI_API_KEY`、`CODEX_API_KEY`、Authorization header 或 auth 文件内容。
- 不把 `auth.json` 贴到工单、聊天、文档或 issue。
- 不在公开 CI、fork PR、共享机器上使用 Codex access token 或长期 API key。
- 不让 repository-controlled code 和 job-level API key 长时间共享同一环境。
- rotation 源目录和目标 `CODEX_HOME` 目录应由部署系统限制权限。

## 决策摘要

`aigameforge-workflow` 的 Codex 主链路选择 `auth.json` / Codex access token，是为了让服务器上的 Codex SDK 以明确的 Codex 用户或 workspace 身份运行，继承 Codex entitlement、workspace 权限、企业治理、会话刷新和审计归属。

API key 仍保留给普通 OpenAI API 调用和隔离的一次性自动化，但不作为 workflow Codex agent 主身份。这个边界能同时降低凭据暴露风险，并让 Codex agent 的身份、额度、审计和失败恢复更符合当前 server workflow 的运行模型。
