# 发布新版本时出现 502 的原因及解决方法

## 现象

发布新版本时，访问页面出现：

```text
502 Bad Gateway
openresty / APISIX
```

这说明浏览器已经访问到了网关，但是网关转发请求到后端容器时失败了。

在这个项目里，可以理解为：

```text
浏览器 -> APISIX 网关 -> Kubernetes Service -> easy-kube Pod
```

502 通常不是浏览器的问题，而是 APISIX 后面没有可用的后端容器，或者后端容器暂时无法处理请求。

## 根本原因

这次问题属于新旧容器衔接问题。

发布新版本时，理想流程应该是：

```text
1. 旧 Pod 继续提供服务
2. 新 Pod 启动
3. 新 Pod 健康检查通过
4. 流量切到新 Pod
5. 旧 Pod 再下线
```

如果配置不合理，实际流程可能变成：

```text
1. 旧 Pod 先被停止
2. 新 Pod 还没有启动完成
3. Service 后面短时间没有可用 Pod
4. APISIX 转发请求失败
5. 用户看到 502
```

所以，核心问题是：发布过程中出现了一段没有可用后端实例的时间窗口。

## 常见触发条件

### 1. 副本数只有 1 个

如果应用只有 1 个 Pod，发布时旧 Pod 被停止，新 Pod 又还没准备好，就很容易出现短暂 502。

建议至少配置 2 个副本：

```yaml
replicas: 2
```

这样发布时，即使一个 Pod 正在替换，另一个 Pod 也可以继续接流量。

### 2. 滚动发布策略不合理

如果 `maxUnavailable` 大于 0，Kubernetes 允许发布过程中减少可用 Pod 数量。

对于线上服务，建议配置：

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0
```

含义：

- `maxSurge: 1`：发布时可以先多启动 1 个新 Pod。
- `maxUnavailable: 0`：发布过程中不允许减少可用 Pod 数量。

这样 Kubernetes 会先启动新 Pod，等新 Pod 可用后，再逐步停止旧 Pod。

### 3. 没有配置 readinessProbe

`readinessProbe` 叫做就绪检查。

它的作用是告诉 Kubernetes：这个 Pod 是否已经准备好接收流量。

如果没有就绪检查，容器刚启动时，Kubernetes 可能会过早把流量转发给它。但此时 Go 服务可能还在初始化，比如连接数据库、加载配置、启动 HTTP 服务。

建议配置：

```yaml
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3
  failureThreshold: 3
```

含义：

- `path: /health`：Kubernetes 访问这个接口判断服务是否就绪。
- `port: 8080`：这里要和 Go 服务实际监听端口一致。
- `initialDelaySeconds: 5`：容器启动后等待 5 秒再开始检查。
- `periodSeconds: 3`：每 3 秒检查一次。
- `failureThreshold: 3`：连续失败 3 次才认为没有准备好。

## Go 服务需要提供健康检查接口

如果配置了 `/health`，Go 服务里也需要真的提供这个接口。

示例：

```go
// /health 是给 Kubernetes 做就绪检查用的接口。
// 只要服务已经可以正常接收请求，就返回 200。
http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
	w.WriteHeader(http.StatusOK)
	_, _ = w.Write([]byte("ok"))
})
```

注意：健康检查接口不要写复杂逻辑。它应该快速返回，避免因为检查接口本身慢导致 Pod 被误判为不可用。

## 旧容器下线前需要优雅退出

发布时，Kubernetes 会给旧容器发送停止信号。

如果 Go 服务收到信号后马上退出，正在处理中的请求会被中断，也可能导致用户看到 502 或请求失败。

更好的做法是：

```text
1. 收到停止信号
2. 停止接收新请求
3. 等待正在处理的请求完成
4. 超时后再退出进程
```

Kubernetes 侧可以配置 `preStop`，让旧 Pod 下线前先等几秒：

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 10"]
terminationGracePeriodSeconds: 30
```

含义：

- `preStop`：容器停止前先执行的动作。
- `sleep 10`：给 APISIX、Service、Endpoints 一点时间把旧 Pod 从流量里摘掉。
- `terminationGracePeriodSeconds: 30`：最多给容器 30 秒做优雅退出。

## 推荐配置组合

解决发布时短暂 502，建议同时做这几件事：

```yaml
replicas: 2

strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0

readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3
  failureThreshold: 3

lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 10"]

terminationGracePeriodSeconds: 30
```

这套配置的目标是：

```text
新 Pod 没准备好，不接流量。
旧 Pod 没被替代前，不下线。
旧 Pod 下线时，不立刻杀进程。
```

## 排查方法

出现 502 时，可以按这个顺序排查：

### 1. 看 Pod 状态

关注是否有 Pod 不是 `Ready`：

```text
Running 但 Ready 不是 1/1：服务启动了，但还没准备好接流量。
CrashLoopBackOff：新版本容器启动失败。
Pending：Pod 还没被调度成功。
```

### 2. 看容器日志

重点查：

```text
端口监听失败
数据库连接失败
配置文件缺失
环境变量缺失
程序 panic
```

### 3. 看服务端口

确认三处端口一致：

```text
Go 服务监听端口
容器端口
Kubernetes Service targetPort
```

例如 Go 服务监听 `8080`，那么健康检查和 Service 的目标端口也应该指向 `8080`。

### 4. 看发布策略

重点确认：

```text
副本数是否至少为 2
maxUnavailable 是否为 0
是否配置 readinessProbe
是否配置 preStop 或优雅退出
```

## 判断是短暂 502 还是持续 502

### 短暂 502

如果 502 只在发布时出现几秒到几十秒，发布完成后恢复，通常是新旧容器衔接问题。

优先解决：

```text
副本数
滚动发布策略
readinessProbe
preStop
Go 服务优雅退出
```

### 持续 502

如果发布后一直 502，通常说明新版本服务本身不可用。

优先排查：

```text
Pod 是否启动失败
容器日志是否报错
服务端口是否配错
镜像是否构建错误
环境变量是否缺失
数据库或外部依赖是否连接失败
```

## 记忆口诀

发布不 502，要做到：

```text
先启动新的，再停止旧的。
新的没 Ready，不许接流量。
旧的要下线，先等请求结束。
副本只有一个，发布必然更容易抖。
```
