**核心题：设计一个企业知识库问答系统**

> 公司有 10 万篇内部文档，包括产品文档、合同、客服记录、运营规则。现在要做一个大模型问答系统，要求：
>
> 1. 用户问问题后，答案必须尽量准确；
> 2. 答案要能引用来源；
> 3. 不能知道就乱编；
> 4. 平均响应时间控制在 3 秒以内；
> 5. 文档每天都会更新；
> 6. 不同用户有不同权限。
>
> 你会怎么设计这个系统？请从数据处理、检索、RAG、模型调用、评估、上线监控几个方面讲。

这题非常好，因为它能一次性测出候选人是不是只会说“向量数据库 + Prompt”，还是懂真实大模型工程。

**重点听这些点**

初级候选人一般会说：

> 文档切 chunk，做 embedding，放进 Milvus/FAISS，用户 query 也 embedding，然后召回文档，塞给 GPT 生成答案。

这只能说明他知道 RAG 基本流程。

中级候选人会补充：

> chunk 策略、metadata、hybrid search、BM25 + 向量召回、rerank、top-k 控制、引用来源、prompt 约束、不知道就拒答、缓存、流式输出、定时更新索引。

这说明做过或至少认真理解过。

高级候选人会主动讲：

> 权限过滤必须在检索前或检索过程中做，不能只靠 prompt；  
> 要有 offline eval 和 online eval；  
> 要区分检索失败、召回成功但生成失败、权限过滤导致无答案、文档本身冲突；  
> 要做 tracing，记录 query、召回文档、rerank 分数、最终答案、用户反馈；  
> 要有 golden set、人工标注集、faithfulness 评估；  
> 延迟上要拆 embedding、retrieval、rerank、generation，各自优化；  
> 文档更新要考虑增量索引、版本号、删除同步；  
> 对高风险问题要返回“基于现有资料无法确认”。

这种基本就是真懂。

**最狠的追问**

问完设计后，你立刻追这个：

> 上线后发现用户反馈“答案看起来很流畅，但经常答错”。你怎么定位问题？

这一下很能试出水平。

好的回答应该能拆成几类：

1. 是不是 query 理解错了？
2. 是不是 chunk 切得不好？
3. 是不是 embedding 召回不到？
4. 是不是 top-k 里有正确文档但 rerank 排低了？
5. 是不是文档召回对了，但 LLM 没有忠实使用？
6. 是不是 prompt 太宽松？
7. 是不是用户问题需要多跳推理，普通 RAG 不够？
8. 是不是数据源本身过期或冲突？

如果他只说“优化 prompt”或者“换更强模型”，基本水平一般。

**再追一个工程味很重的**

> 如果用户问：“根据我司最新报销制度，出差打车能不能报销？”但系统召回了三份互相矛盾的制度文档，你怎么办？

高级回答会提到：

- 文档版本号
- 生效日期
- 权威来源优先级
- 冲突检测
- 答案中暴露不确定性
- 返回引用而不是强行下结论
- 必要时转人工或提示“资料冲突”

**最后一个杀招**

> 什么时候你不会用 Agent？

这题很妙。很多包装型候选人会什么都说 Agent。

靠谱的人会说：

> 简单、确定、低分支的流程不需要 Agent；  
> 规则明确的任务优先用普通 pipeline；  
> Agent 适合多步骤、不确定、需要工具调用和动态决策的场景；  
> 但 Agent 会带来延迟、成本、不可控、评估困难，所以要谨慎用。

这个回答比“Agent 能自动规划任务”靠谱很多。

你可以就用这一套：**一个系统设计题 + 三个追问**。基本 20 分钟就能判断他是会背概念、做过 demo，还是有真实工程能力。
